# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Single-page personal portfolio for Tyson Farley. A Vue 2 + Vuetify 2 SPA bundled
with Webpack 5. The site renders a career timeline; the most common change is
adding new timeline entries to the existing "Period" components.

## Commands

Run `npm` scripts through the **Bash tool** — PowerShell's execution policy blocks
the `.ps1` shims on this machine.

- `npm run build` — Webpack build (defaults to `mode: development`), outputs to `dist/`.
- `npm run dev` / `npm run prod` — build forcing development / production mode.
- `npm run watch` (alias `npm start`) — rebuild on change.
- `npm test` — Jest (jsdom) over all `*.test.js`.
- Single test file: `npx jest src/components/App/App.test.js`
- Single test by name: `npm test -- -t "renders passed content"`
- Lint (no npm script defined): `npx eslint "src/**/*.{js,vue}"` — config is `eslint:recommended` with the `@babel/eslint-parser`.

There is no dev server; preview by building and opening `dist/index.html` (the
HTML is generated by `html-webpack-plugin`, assets in `resource/` are copied to
`dist/resource/`).

## Deployment (GitHub Pages)

The site is served from the `gh-pages` branch via the `git subtree push` technique
(per https://gist.github.com/cobyism/4730490). **`dist/` is committed to the repo and
is exactly what gets served** — it is *not* a throwaway build artifact (only
`node_modules` is gitignored). So the deploy flow is:

1. `npm run prod` — rebuild `dist/` (it's `webpack --clean`, so it's fully regenerated).
2. Commit the updated `dist/` (along with the source changes).
3. Push the subtree to `gh-pages`:
   ```bash
   git push origin :gh-pages && git subtree push --prefix dist origin gh-pages
   ```
   (deletes the stale remote branch first, then republishes `dist/` as a fresh tree).

Consequence: after any source change, a stale `dist/` will keep serving the old build.
Always rebuild + commit `dist/` before deploying.

## Module resolution (important gotcha)

Both Webpack (`resolve.modules`) and Jest (`moduleDirectories`) add these dirs to
the resolution roots: `src`, `src/components`, `src/pages`, `src/plugins`,
`src/util`. Imports therefore use **bare, root-relative paths**, not `./` /`../`:

```js
import Home from 'Home/Home.vue';                       // from src/pages
import TyApp from 'App/App.vue';                         // from src/components
import { mount } from 'init.js';                         // from src/util
import System from 'stores/System/System.js';           // from src
import { createShallowMountFactory } from 'util/test-helpers.js';
```

When adding a file, place it so these roots can resolve its imports. CSS/SCSS is
stubbed in tests via `moduleNameMapper` → `mock.js`.

## Architecture

- **Entry & mount**: Webpack `entry: './src'` → [src/index.js](src/index.js) calls
  `mount(Home, HomeStore)` from [src/util/init.js](src/util/init.js). `init.js` is the
  one place global plugins are wired (Vuex, Vuelidate + vuelidate-error-extractor
  bound to `FormGroup`, the global `errorHandler`/`window.onerror`). `mount()`
  injects an `#app` div and instantiates the root Vue. (`Home.entry.js` is empty/legacy.)

- **Component wrappers (`Ty*` convention)**: App-owned components wrap Vuetify
  primitives and are prefixed `Ty` (`TyApp`, `TyTimeline`, `TyTimelineItem`,
  `TyButton`, `TyIcon`, `TyImage`, `TyLink`, `TyForm`, `TyTextField`, `TyFormGroup`,
  `TyRecaptcha`, `TyContainer`, snackbar/dialog). Build on these wrappers rather than
  using `v-*` Vuetify components directly in pages.

- **Page composition**: [Home.vue](src/pages/Home/Home.vue) → `TyApp` (app bar /
  footer / notification slots) → `SystemNotificationHub` + `PersonalHistory`.

- **Portfolio content lives in Period components**:
  [src/components/Period/<Employer>/<Employer>.vue](src/components/Period/) (e.g.
  `Domo`, `StateFoodSafety`). Each is a `TyTimeline` (with `title`, `website`,
  `colors`) containing a list of `TyTimelineItem`s. A timeline item is just:

  ```html
  <TyTimelineItem :date="new Date('May 26, 2022')" title="Short Title">
      Free-text description of the milestone.
  </TyTimelineItem>
  ```

  `TyTimelineItem` auto-formats the date to "Month Year", auto-collapses cards taller
  than `MAX_CARD_HEIGHT` (180px) behind an expand arrow, and gets its `color`/`dense`
  assigned by the parent `TyTimeline`. **Adding career entries = adding
  `TyTimelineItem`s to the relevant Period component** (newest existing examples are in
  `Domo.vue`); register new Period components in
  [PersonalHistory.vue](src/components/PersonalHistory/PersonalHistory.vue).

- **State (Vuex)**: One namespaced module, `System`
  ([src/stores/System/](src/stores/System/)), holding `alerts`, `announcements`,
  `messages`. State keys, getters, and mutations are each declared as **exported
  string constants** in separate files (`SystemStateKeys.js`, `SystemGetters.js`,
  `SystemMutations.js`) and used as computed property keys (`[GET_ALERTS]`). Add new
  store members by following this constants-in-separate-files pattern. Page stores
  compose modules: [Home.store.js](src/pages/Home/Home.store.js).

- **Notifications**: Three surfaces — `AlertSnackbar` (timed), `MessageDialog`
  (modal), `AnnouncementBar` (dismissible). `SystemNotificationHub` reads them from
  the store and renders one instance per enabled type. Types
  (`success`/`info`/`warning`/`error`) and their icon/title/color maps come from the
  `Notification` mixin factory ([src/mixins/Notification/Notification.js](src/mixins/Notification/Notification.js)).

- **Mixins**: `Model` ([src/mixins/Model/Model.js](src/mixins/Model/Model.js))
  implements the standard `v-model` ↔ `localValue` bridge (emits `input`, plus
  `open`/`close`); reuse it for any component exposing `v-model`.

- **Theme**: [src/plugins/vuetify.js](src/plugins/vuetify.js) defines a custom light
  theme keyed to elemental color names (`fireRed`, `fireOrange`, `fireYellow`,
  `windGray`, `rainBlue`, `leafGreen`, `rockBrown`) which also alias the semantic
  Vuetify slots (`primary`, `error`, etc.). Use these names for `colors`/`color` props.

## Testing

- Tests are colocated as `<Component>.test.js` and use
  `createShallowMountFactory(Component)` from
  [src/util/test-helpers.js](src/util/test-helpers.js), which wires a local Vue with
  Vuex + the real Vuetify instance. Existing tests mostly assert slot rendering.
- `jest.setup.js` registers Vuetify globally; SCSS/CSS imports resolve to `mock.js`.

## Conventions

- `.editorconfig` is authoritative: **tabs, indent size 4, LF, UTF-8, final newline,
  trim trailing whitespace.** Match this in all files.
- Components set `name: 'Ty...'` and use `inheritAttrs: false` with
  `v-bind="$attrs"` / `v-on="$listeners"` to forward to the wrapped Vuetify primitive.
